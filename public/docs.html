🧠 What is Context Engineering?

Context Engineering is the discipline of designing structured configuration systems that transform generic AI coding agents into specialized, opinionated architects for your specific project.

Without a Blueprint, an AI agent produces generic code. With a Blueprint, it produces code that follows your exact stack, your architecture patterns, your security policies, and your quality standards — consistently, every time.

Think of it like this: a fresh AI agent is a brilliant intern on Day 1 — talented but with zero context about your project. A Blueprint is the onboarding package that makes them productive immediately: it tells them which technologies you use, how your code is organized, what your naming conventions are, and what shortcuts are forbidden.

Why does this matter?

AI coding agents (Google Antigravity, GitHub Copilot, Cursor) are only as good as the context you give them. Most developers provide minimal context — a file or two. Context Engineering provides systematic, layered context that covers architecture, security, testing, deployment, and coding style. The result: dramatically better code output with fewer iterations. An AI reading a Blueprint stops being an assistant and becomes a Senior Architect executing your exact standards.

⚖️ The 6 Fundamental Principles

Every Blueprint generated by the Compiler adheres to these six principles. They are the foundation of Context Engineering methodology.

1. Enforcement > Guidelines

Every rule must have a technical enforcement mechanism. Saying "use TypeScript strict mode" is a guideline. Adding "strict": true in tsconfig.json plus a CI step that fails on type errors — that's enforcement. Blueprints don't suggest; they enforce.

2. Progressive Disclosure

Load context only when needed. Skills are loaded on-demand, not always-on. The agent doesn't need to know how to deploy when it's writing a database migration. This prevents context window pollution and keeps the agent focused.

3. Plan First, Code Second

The agent must never write code without an approved implementation plan. The Plan Mode Contract requires: create plan → present to user → wait for approval → only then implement. This prevents wasted effort and ensures alignment.

4. Compilable Snippets

Every code example in a Blueprint must be syntactically valid and copy-pasteable. No pseudo-code, no [TODO: fill in], no hand-waving. If a snippet is in the Blueprint, it must work when pasted into the project.

5. Total Internal Coherence

No rule can contradict another. If the rules say "use Drizzle ORM" then the skills, workflows, and code templates must all reference Drizzle — never Prisma. All file paths, tool names, and references must exist and be consistent across all layers.

6. Global/Project Separation

Global configuration (~/.gemini/GEMINI.md) contains only the thinking protocol and git discipline. Language preferences, identity, plan mode, security rules — everything else lives at the project level. This separation makes the framework reusable across any project.

🏗️ The 6-Layer Architecture

A Blueprint is organized into 6 distinct layers, each with a specific purpose. Layers are loaded in order, with higher layers building on lower ones.

Layer Name Location Purpose
L1 Global Config ~/.gemini/ Universal rules — thinking protocol + git discipline
L2 Workspace Rules .antigravity/rules.md Project-specific stack, architecture, conventions
L3 Skills .agent/skills/ On-demand competencies with templates
L4 Workflows .agent/workflows/ Step-by-step procedures enforcing order
L5 Context .context/ Architecture Decision Records + style guide
L6 Enforcement Map Embedded in L2 Maps rules → tools → failure modes

Layer 1 Global Configuration

The global layer lives in ~/.gemini/ and is identical for all projects. It contains exactly two things:

GEMINI.md (~15 lines)

Thinking Protocol: Instructions for how the agent should reason — think step-by-step, consider trade-offs, explain decisions.

Git Discipline: Commit frequency, message format, branch naming conventions.

settings.json

Defines coreTools (terminal, filesystem, web), excludeTools (dangerous commands like rm -rf, sudo), and mcpServers for external integrations.

⚠️ Critical design decision: Language, identity, and security rules do NOT belong in L1. They are project-specific and live in L2. This is what makes the framework truly reusable across domains.

Layer 2 Workspace Rules

The heart of the Blueprint. A single file (.antigravity/rules.md) containing 13 precisely defined sections, typically 400-500 lines.

# Section What It Defines
1 Language & Identity Conversation language, code language, agent's senior role
2 Plan Mode Contract Implementation plan template with mandatory STOP-and-wait
3 Terminal Restrictions Project-specific blocked commands beyond global excludes
4 Code Quality Standards Language-specific: strict types, no any, validation rules
5 Security First Framework-specific security checklist
6 Project Mission One-line product description
7 Tech Stack LOCKED Technology table with rationale — agent cannot deviate
8 Architecture Pattern Core pattern (e.g., Clean Architecture) with enforcement
9 Data Isolation Multi-tenancy, access control, Row Level Security
10 Folder Structure Complete directory tree the agent must follow
11 Data Fetching Server components, client components, caching strategy
12 Frontend Conventions UI patterns, component structure, accessibility
13 Testing Strategy Unit/integration/E2E split, coverage targets, tools
14 Environment Config Env variables, secrets management, deployment targets

Layer 3 Skills

Skills are modular competencies stored as individual Markdown files in .agent/skills/. Each skill teaches the agent HOW to perform a specific task. Skills are loaded on-demand — the agent reads them only when needed.

Skill Structure

## Goal
What this skill achieves (one paragraph)

## Instructions
1. Step-by-step procedure
2. With specific commands and checks
3. Including error handling

## Template
```typescript
// Compilable code snippet ready to paste
export function exampleFunction(): Result {
  // Real implementation, not pseudo-code
}
```

## Constraints
- What NOT to do
- Edge cases to handle
- Performance requirements

A typical Blueprint generates 5-8 skills tailored to the domain. Examples: new-api-endpoint.md, database-migration.md, auth-flow.md, component-creation.md.

Layer 4 Workflows

Workflows are step-by-step procedures that enforce the correct order of operations. Unlike Skills (which teach HOW), Workflows define WHEN and IN WHAT ORDER.

Universal Workflows (included in every Blueprint)

new-feature.md — The mandatory sequence: Plan → Data Model → Business Logic → UI → Tests → Verify. The agent cannot skip steps.

deploy-check.md — Pre-deployment checklist: lint → typecheck → test → build → security audit. Must all pass before deployment.

Layer 5 Context Files

Context files live in .context/ and provide the WHY behind technical decisions.

architecture.md — Architecture Decision Records (ADR)

5-6 formal records documenting why each major technology was chosen. Each ADR contains: Title, Status (accepted/deprecated), Context (what problem we faced), Decision (what we chose), Alternatives Considered, Consequences (trade-offs accepted).

coding_style.md — Coding Conventions

Naming conventions (camelCase vs snake_case), file naming patterns, import ordering, comment style, error handling patterns. Specific to the chosen stack.

Layer 6 Enforcement Map

The Enforcement Map connects every critical rule to its enforcement mechanism. It answers three questions for each rule:

Question Example
What tool enforces it? ESLint, TypeScript compiler, CI pipeline
When does it run? On save, pre-commit, CI, deploy
What happens if it fails? Build fails, PR blocked, deployment aborted
This is what separates Context Engineering from "just writing good prompts." Every rule has teeth — a mechanism that prevents violations rather than merely discouraging them.

Quality Scoring

Every generated file is scored on a 0-100 scale across multiple dimensions:

Dimension Weight Measures
Completeness 30% All expected sections present, minimum line count met
Specificity 25% References exact stack technologies (not generic), no placeholder text
Structure 20% Proper markdown formatting, hierarchy, consistent style
Actionability 15% Compilable code snippets, concrete patterns (not abstract advice)
Coherence 10% Internal consistency, correct cross-references between files
A score of 75+ (Grade B) is the default threshold for automatic refinement. Below this, the engine automatically regenerates with targeted fix instructions.

Blueprint Rigor Levels

Control how strict the generated Blueprint is:

🏛️ Strict — Enterprise / Production

Full planning required before every task. 90%+ test coverage. Complete OWASP security checklist. Architecture Decision Records for every choice. For production systems and regulated industries.

⚖️ Balanced — Recommended

Complexity-gated planning (simple tasks skip planning). 80%+ test coverage. Essential security controls. Best for most production projects.

🚀 Rapid — Prototype / MVP

Minimal planning. Critical-path testing only. Ship fast. For hackathons, demos, and quick prototypes where speed matters more than rigor.

📚 Complete Glossary

An exhaustive, alphabetical reference of every term, concept, and technology used in the Blueprint Compiler ecosystem.

ADR (Architecture Decision Record)
A formal document recording a significant architectural decision: the context, the decision itself, alternatives considered, and consequences. Part of Layer 5. ADRs are immutable — when a decision changes, a new ADR supersedes the old one rather than editing it.
Agent
An AI coding assistant (e.g., Google Antigravity, GitHub Copilot, Cursor) that reads your code, understands instructions, and generates or modifies code. Without a Blueprint, an agent has no project-specific context and produces generic output.
API Layer
The communication protocol between frontend and backend. Options include tRPC (type-safe, zero documentation needed), GraphQL (flexible queries for complex data), REST (universally understood), and Hono (edge-first, ultra-lightweight).
Authentication
The system that verifies user identity (login). Options range from self-hosted (NextAuth.js — full control, free) to managed services (Clerk — beautiful pre-built UI) to enterprise-grade (Auth0 — SAML, LDAP, SSO support).
BaaS (Backend as a Service)
A cloud service providing pre-built backend functionality: database, authentication, file storage, serverless functions. Examples: Supabase (open-source, PostgreSQL), Firebase (Google, NoSQL), Appwrite (self-hosted). Eliminates the need to build a custom backend.
Blueprint
A complete set of configuration files (typically 5) that transforms a generic AI agent into a specialized architect for your specific project. A Blueprint defines your stack, architecture, conventions, security policies, and coding standards. It consists of Layer 2-6 files.
Blueprint Compiler
The web application that generates Blueprints. A 4-step wizard: Domain → Stack → Project → Generate. 100% client-side React, zero backend. Calls the Anthropic Claude API directly from the browser using the user's own API key.
CDN (Content Delivery Network)
A global network of servers that cache and serve content close to users. For e-commerce: Cloudinary (image transformation), Imgix (URL-based processing), Vercel Image (zero-config with Next.js), Cloudflare Images (competitive pricing).
CI/CD
Continuous Integration / Continuous Deployment. Automated pipelines that test, build, and deploy code. The deploy-check workflow (Layer 4) defines the exact steps: lint → typecheck → test → build → security audit.
Context Engineering
The discipline of designing structured, layered configuration systems for AI coding agents. Goes beyond prompt engineering (which focuses on individual prompts) by creating systematic, reusable, enforceable configuration that persists across sessions.
Context Window
The maximum amount of text an AI model can process at once. Progressive Disclosure (Principle 2) prevents filling the context window with irrelevant information, keeping the agent focused on the current task.
Custom Domain
The 6th domain option in the Compiler. For projects that don't fit predefined categories. Users describe their domain and stack manually. Examples: IoT, game dev, AI/ML platforms, DevOps tooling.
Data Platform
One of the 6 domains. Covers ETL/ELT pipelines, data warehouses, transformation frameworks, and visualization tools. Key technologies: Airflow/Dagster (orchestration), BigQuery/Snowflake (warehouse), dbt (transformation), Metabase/Grafana (visualization).
Domain
A category of software project with distinct architectural patterns, technology choices, and challenges. The Compiler supports 6 domains: SaaS B2B, WordPress, E-commerce Headless, Mobile App, Data Platform, and Custom.
E-commerce Headless
One of the 6 domains. Architecture where the shopping frontend (storefront) is decoupled from the commerce backend (products, cart, checkout). Key technologies: Shopify+Hydrogen, Medusa.js (open-source), Algolia (search), Stripe (payments).
Enforcement Map
Layer 6 of the Blueprint. Maps every critical rule to: (1) what tool enforces it, (2) when it runs, (3) what happens if it fails. Ensures rules have technical enforcement, not just documentation.
ETL / ELT
Extract, Transform, Load / Extract, Load, Transform. Data pipeline patterns. ETL transforms data before loading into the warehouse. ELT loads raw data first, then transforms inside the warehouse. Modern stacks favor ELT with tools like dbt.
Global Config
Layer 1 of the Blueprint. Lives in ~/.gemini/. Contains only the thinking protocol and git discipline. Identical across all projects — never includes project-specific settings.
Headless
An architecture where the frontend presentation layer is completely separated from the backend business logic. The frontend communicates with the backend only through APIs. Common in e-commerce (Shopify Headless) and CMS (Headless WordPress).
Layer
One of the 6 structural levels of a Blueprint. Layers are loaded in order, with each adding context: L1 (global) → L2 (rules) → L3 (skills) → L4 (workflows) → L5 (context) → L6 (enforcement).
Meta-Template
The master system prompt (~1,052 lines) that defines HOW to generate Blueprints. It is the "Blueprint that generates Blueprints." Embedded in the Compiler as a system prompt sent to Claude. This is the core intellectual property of the project.
Mobile App
One of the 6 domains. Covers cross-platform and native development. Key technologies: React Native+Expo (cross-platform), Flutter (Google, Dart), Zustand (state), Supabase/Firebase (backend), WatermelonDB (offline storage).
Multi-tenancy
An architecture where multiple organizations share the same application and database infrastructure, but each sees only their own data. The SaaS B2B Blueprint enforces this through PostgreSQL Row Level Security and middleware-level tenant validation.
ORM (Object-Relational Mapping)
A library that lets you interact with databases using your programming language instead of raw SQL. Options: Drizzle (lightweight, SQL-like), Prisma (abstracted, own schema language), TypeORM (decorator-based, enterprise), Mongoose (MongoDB documents).
Plan Mode Contract
Section 2 of Layer 2 (Workspace Rules). Requires the agent to create an implementation plan, present it to the user, and STOP until approved. Prevents the agent from writing code before confirming the approach. Enforces Principle 3 (Plan First, Code Second).
Progressive Disclosure
Principle 2 of Context Engineering. Only load context into the agent's attention when it's relevant to the current task. Skills are on-demand, workflows are invoked only when needed. Prevents context window pollution.
PROMPT_START
The entry point file generated by the Compiler. Contains a project overview, file tree, and quick start commands. The agent reads this first when starting work on a project. Typically 60-80 lines.
Row Level Security (RLS)
A PostgreSQL feature that restricts which rows a user can access based on policies. Used in SaaS multi-tenancy: a policy like tenant_id = current_user_tenant() ensures each company sees only its own data, enforced at the database level.
SaaS B2B
One of the 6 domains. Software as a Service sold to businesses. Multi-tenant architecture. Key technologies: Next.js (frontend), tRPC (API), Drizzle+PostgreSQL (database), NextAuth.js (auth), Stripe (billing), Vercel (hosting).
Skill
A Layer 3 file that teaches the agent a specific competency. Each skill has: Goal (what), Instructions (how), Template (compilable code), Constraints (boundaries). By default, the Compiler now automatically injects community-vetted Official Skills from the Enterprise Skill Registry saving thousands of tokens and enforcing perfect architectural patterns.
Enterprise Skill Registry
The public repository of official, heavily vetted Markdown skills (e.g. Next.js Routing, Supabase Security). The Compiler's AI engine automatically selects the perfect skills from this registry based on your tech stack, injecting them into your Blueprint without generating token-heavy boilerplate.
Stack
The complete set of technology choices for a project. Each domain has 5-6 categories with multiple options. The Stack is "LOCKED" in the Blueprint (Section 7 of L2) — the agent cannot substitute technologies without explicit approval.
System Prompt
Initial instructions given to an AI model before any user messages. The Meta-Template serves as the system prompt when the Compiler generates Blueprint files. It tells Claude how to structure the output according to Context Engineering methodology.
Workflow
A Layer 4 file defining a step-by-step procedure. Unlike Skills (which teach HOW to do something), Workflows define WHEN and IN WHAT ORDER steps must happen. Universal workflows: new-feature.md (Plan→Data→Logic→UI→Test→Verify), deploy-check.md (lint→typecheck→test→build→audit).
WordPress
One of the 6 domains. World's most popular CMS (43% of websites). Key technologies: WordPress 6.7+ (latest), Gutenberg Block Editor (native, zero overhead), WooCommerce (e-commerce), Yoast SEO, WP Rocket+Redis (caching), Kinsta (hosting).
Workspace Rules
Layer 2 of the Blueprint. The single most important file (.antigravity/rules.md). Contains 13-14 sections defining everything project-specific: language, plan mode, code quality, security, stack, architecture, folder structure, testing, and more. Typically 400-500 lines.

❓ Frequently Asked Questions

Do I need to know how to code to use the Blueprint Compiler?

Not necessarily. The Compiler generates configuration files that tell an AI agent how to code for you. However, basic understanding of your chosen technologies helps you make better stack choices. The contextual guides next to each option are designed to educate you during the process.

Which AI agent works with these Blueprints?

Blueprints are primarily designed for Google Antigravity, but the concepts are transferable to any AI coding agent that supports custom instructions or system prompts (Cursor, GitHub Copilot, etc.).

How much does it cost?

The Compiler itself is free. You need an Anthropic API key to generate files. Each file costs $0.02-0.05 in API usage. A complete Blueprint (5 files) costs approximately $0.15-0.25.

Is my API key safe?

Yes. Your API key is stored in sessionStorage (browser tab only — disappears when you close the tab). API calls go directly from your browser to Anthropic — the key never touches any intermediate server.

Can I regenerate individual files?

Yes. Each of the 5 files can be generated and regenerated independently. If the Skills output doesn't match your needs, regenerate just that file without affecting the others.

What if my project doesn't fit any domain?

Choose Custom Domain. You'll describe your domain and technology stack manually. The Compiler will generate a Blueprint tailored to your specific needs, following the same 6-Layer methodology.

Can I modify the generated files?

Absolutely — and you should. Generated Blueprints are a strong starting point (rated 8.5/10 by an independent reviewer). Customize them to match your team's exact conventions, add project-specific skills, and evolve them as your project grows.
